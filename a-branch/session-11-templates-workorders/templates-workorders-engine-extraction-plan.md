# Templates + Work Orders — Engine Extraction & Module Sweep

> **Status: APPROVED & LOCKED** — 2026-04-29
> Do not edit this file post-approval. Updates go to the execution file.

## Context

Templates and work-orders are the first two modules to be fully migrated off `apps/web/modules/shared/engines/`. The engine isn't a junk drawer — about 40% of it is load-bearing infrastructure (save/discard/dirty controllers, scaffolds, multi-section panel orchestration, mutation telemetry, record-entry navigation). The problem is that all of it lives in one folder, so consumers depend on the path instead of the concern. Once the load-bearing concerns are extracted into peer directories under `apps/web/`, modules can layer cleanly onto them, and the engine reduces to dead code that can be deleted with grep proof.

Two acute symptoms motivate this sweep:
- **Templates list view** still uses `useConfiguredTableState` (engine, client-side filter + grouping rendering). It can't scale to thousands of rows.
- **Work-orders material-items section** is hand-rolled (raw `<table>` + raw `<select>`/`<input>`), which bypasses `useSingleSectionRecordController`'s dirty-state plumbing. That's why clicking save doesn't refresh the section. The visual layout is good and stays — the implementation gets rebuilt on the canonical Templates pattern (`Grid` + canonical cells + section controller).

By sweep end: templates and WO import nothing from `modules/shared/engines/`; new peer dirs `hooks/`, `scaffolds/`, plus new `components/` sub-buckets are established; server-side filter joins sort/search/paginate/grouping in `useServerListController`; engine files with zero remaining consumers are deleted.

## Decisions Locked

| # | Decision | Choice |
|---|---|---|
| 1 | `controllers/` vs `hooks/` naming | Keep both. `controllers/` = owns state for a feature (list, record section). `hooks/` = lifecycle/utility/cross-cutting. Section-controller hooks (`useSingleSectionRecordController`, etc.) → `controllers/record/`. |
| 2 | `scaffolds/` placement | Top-level peer (`apps/web/scaffolds/`). Page-shape contracts, not generic primitives. |
| 3 | Engine cells vs canonical cells | Not duplicates. Engine cells are layout wrappers without input contracts; `components/cells/` are full cell contracts. Engine cells get **deleted** after consumers migrate, not merged. |
| 4 | Grouping in templates list | Drop in this sweep. Aligns with the planned grouping-tool separation. |
| 5 | `record-view/shell/` audit | Folded into this sweep (not deferred). |
| 6 | Hand-rolled primary/MI sections | None survive. WO MI rebuilt on Templates MI pattern. |
| 7 | Server-side filters | Extend `useServerListController` + extract canonical `features/filter/` this sweep. |
| 8 | `record-primary-fields.tsx` | Extract as-is to `components/sections/structure/`. Convergence with `LayoutGrid`/`FieldSection` deferred. |
| 9 | Migration order | Extract-then-sweep, WO before templates. |
| 10 | `record-view/forms/` | Delete after templates primary moves to canonical fields/cells. |

## Phase 1 — Extract Engine Concerns to New Peer Dirs

No consumer changes yet. Engine paths stay alive; new peer dirs become the canonical home. Dual-path period is brief but expected.

**Sub-phase 1a — server/telemetry**
- Move `engines/common/application/` → `apps/web/server/telemetry/`. Update internal references inside the moved files.
- Engine files become re-export shims pointing at `@/server/telemetry/*` (tracked as candidates for deletion in Phase 4).

**Sub-phase 1b — transport already exists, no work**
- `@/transport/{http,mutation,client-errors}` already exist. Engine files (`engines/common/transport/*`) are already shims.

**Sub-phase 1c — hooks/navigation + hooks/record**
- Move `engines/common/record-entry/` → `apps/web/hooks/navigation/`.
- Move `engines/record-view/client/hooks/` → `apps/web/hooks/record/`.
- Move `engines/common/feedback/{confirm-delete,notices,feedback-states}` → `apps/web/components/{dialogs/confirm-delete, feedback/notices, feedback/states}`.

**Sub-phase 1d — controllers/record**
- Move `engines/record-view/client/controllers/` → `apps/web/controllers/record/`.
- Move `engines/record-view/client/utils/` → `apps/web/controllers/record/utils/` (or fold into the relevant controller files).

**Sub-phase 1e — scaffolds + components/panels + components/sections**
- Move `engines/record-view/client/scaffolds/` → `apps/web/scaffolds/`.
- Move `engines/record-view/shell/{record-detail-page-shell,record-primary-header,record-action-buttons,record-panel-footer,index}` → split: page-level pieces to `scaffolds/`, button/footer to `components/panels/`.
- **Drop now**: `record-view/shell/{primary-record-panel,record-options-menu,record-panel-width}` — zero module consumers, zero internal usage. Inline the `max-w-none` constants into `record-detail-page-shell` at move time.
- Move `engines/record-view/panel/` → `apps/web/components/panels/`.
- Move `engines/record-view/sections/{panels,metrics,structure}/` → `apps/web/components/sections/{panels,metrics,structure}/`.
- Move `engines/record-view/shell/record-primary-fields.tsx` → `apps/web/components/sections/structure/record-primary-fields.tsx` **as-is**, no API change.
- Move `engines/record-view/feedback/` → `apps/web/components/feedback/{errors,notices}/`.

**Sub-phase 1f — composites**
- Move `modules/shared/property-fields/` → `apps/web/components/composites/property-fields/`.

**Sub-phase 1g — server-side filters in `useServerListController`**
- Extend `apps/web/controllers/list-view/useServerListController` with filter state + URL binding (mirror sort/search wiring).
- Extract engine filter chrome to a new canonical `apps/web/components/features/filter/` (FilterControl, filter chip strip). Keep API surface narrow: `filters`, `onFilterChange`, `filterableFields`.
- Server contracts: extend the `listX(...)` request type used by templates+WO with a `filters` parameter; pipe through to the read repository. Reuse the imports module's existing partial setup as the reference pattern.

**Phase 1 acceptance:** every new peer dir exists with the moved files; `tsc` passes; engine paths still resolve via shims; no module yet imports from a new peer dir.

## Phase 2 — Sweep Work-Orders Consumers

WO is partially migrated (list view already on `useServerListController`, primary section already canonical). Two big items: MI rebuild, plus path swaps.

**2a — MI section rebuild (the big one)**
- Replace WO MI's hand-rolled `<table>` + raw inputs with: `Grid` (from `apps/web/components/grid/`) + canonical cells (`apps/web/components/cells/`) + `useSingleSectionRecordController` from `apps/web/controllers/record/`.
- Adopt Templates MI as the structural reference. Preserve the visual density/layout WO MI has today (spacing tokens carry over).
- Add expandable-row capability for cut logs — WO MI is gaining a per-row dropdown to expand cut logs (per CLAUDE.md sweep scope). Use `apps/web/components/grid/expandable-rows/`. The existing pattern in `webb/app/components-smoke` is the reference.
- Use cases: WO MI must support saving pending cut logs, finalizing cut logs, and material-item updates. Worker/relay/outbox layering follows the canonical pattern (see imports).
- Verify: clicking save now refreshes the section (the original symptom).

**2b — Path swaps for WO record view**
- Swap engine imports → new peer dirs: `RecordDetailClientScaffold`, `RecordSingleSectionPanel`, `RecordPanelRenderer`, section panels (`RecordFieldSection`, `RecordPrimarySectionInstance`), feedback components, `record-primary-fields.tsx`.
- Swap `engines/common/record-entry` → `@/hooks/navigation`.
- Swap `engines/record-view/client/hooks/use-record-notices` → `@/hooks/record/use-record-notices`.
- Swap `engines/common/application/{mutation-telemetry,loader-timing}` → `@/server/telemetry/*`.
- Swap `engines/common/feedback/confirm-delete` → `@/components/dialogs/confirm-delete`.
- Swap `engines/common/transport/*` → `@/transport/*` (already extracted; this is just an import path cleanup).

**2c — Files / template-sync UI hooks**
- Add the third "Files" section to WO record view (work order row + material items data files; bucket-stored). Per CLAUDE.md scope. Use `apps/web/components/sections/panels/RecordItemSection` for layout.
- Hook up the "create work order by syncing a template" icon in the top-right app shell. Imports the canonical use case path (Domain → Data → Application → API).

**Phase 2 acceptance:** `rg "modules/shared/engines" apps/web/modules/work-orders` returns zero. WO record view + list view + MI section all work end-to-end, save refreshes correctly.

## Phase 3 — Sweep Templates Consumers

Templates is further from canonical: still on engine list controller, primary section still imports from `record-view/forms/`.

**3a — List view migration**
- Replace `useConfiguredTableState` + `useTableControls` with `useServerListController` (fetch mode, mirror WO + imports patterns).
- Drop client-side grouping rendering. Drop grouping toggle entirely (per Decision 4).
- Wire the new server-side filter from Phase 1g.
- Add server-side `listTemplatesRequest` (thin wrapper over canonical read; paginated + filterable).
- `disableClientPagination: true` paths get removed entirely — pagination is fully server-driven.

**3b — Primary section to canonical fields/cells**
- Templates' `TemplatePrimaryFieldsSection` currently imports from `engines/record-view/forms/` (`RecordFormField`, etc.). Rewrite to use `apps/web/components/fields/` + `apps/web/components/cells/` (mirror WO's already-migrated primary section).
- Keep the `RecordPrimarySectionInstance` wrapper (still canonical).

**3c — MI section path swaps**
- Templates MI is already on `Grid` + canonical cells. Just swap imports for the moved engine pieces (panels, sections, feedback, record-primary-fields, scaffolds).

**3d — Remaining path swaps**
- Same swap list as Phase 2b applied to the templates module tree.

**Phase 3 acceptance:** `rg "modules/shared/engines" apps/web/modules/templates` returns zero. `rg "engines/record-view/forms" apps/web` returns zero outside the engine tree itself. Templates list paginates server-side, filters server-side.
