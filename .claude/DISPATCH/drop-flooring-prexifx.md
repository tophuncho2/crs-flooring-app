## ⚑ Flags — decisions to make / potential gaps
- ⚑ **`@@map` RE-POINT, not drop.** The mission was originally framed as "drop the now-redundant `@@map`" — that premise is WRONG for this repo. Prisma's default table for model `Category` is `Category` (PascalCase); to keep the existing snake_case table `category` you MUST re-point `@@map("category")`. The canonical `FlooringTemplate`→`Template` rename kept `@@map("template")`. Re-point all 14. Dropping them silently repoints Prisma at PascalCase tables that don't exist. (Recommend re-point — flagged only because the mission framed it as a drop.)
- ⚑ **`FlooringTimeOfDay` has ZERO hand-written code refs** (schema + generated client only; used by field `timeOfDay` on WorkOrder, schema line 650). Rename it (`→TimeOfDay`, consistent, cheap) vs drop the unused enum entirely (drop needs `DROP TYPE` + a decision about the `timeOfDay` column). Recommend rename for consistency; surface to user.
- ⚑ **Outbox `aggregateType: "FlooringImportEntry"` persisted event-contract literal.** `packages/application/src/imports/staged-inventory-rows/mark-staged-rows-for-import.ts:101` (+ test `packages/application/tests/imports/mark-staged-rows-for-import.test.ts:162`). This is a persisted event-envelope discriminator written into `queue_outbox_event` rows, NOT a type or table name. Renaming it must coordinate with any consumer matching on the literal AND in-flight outbox rows already carrying the old value. **Surface, do not auto-rename** — freezing it is defensible.
- ⚑ **The 7 `@relation("Flooring…")` disambiguator strings** (schema lines 230/250, 231/263, 232/302, 233/453, 234/367, 235/396, 237/717) are cosmetic — not persisted, zero accessor impact, zero code blast either way. The canonical commit DID rename its analog. Rename-for-honesty vs leave.
- ⚑ **The 2 Product back-relation FIELD names** (schema lines 273/274: `FlooringImportStagedInventoryRow FlooringImportStagedInventoryRow[]` and `FlooringImportStagedInventoryFilterRow FlooringImportStagedInventoryFilterRow[]` — Prisma auto-named the field after the model). No DB column; grep confirms no code reads `product.FlooringImportStagedInventoryRow`. The TYPE must change; the FIELD name is cosmetic (rename to de-prefixed / honest vs leave the odd names).
- ⚑ **Scripted vs manual replacement — MUST enforce longest-name-first + word-boundary order.** A naive global `s/Flooring//` corrupts nested tokens (turns `FlooringImportStagedInventoryFilterRow` into a wrong result if `FlooringImportStagedInventoryRow` runs first) AND hits the out-of-scope nav/page symbols (see Scope Out / §3c). Use the ordered list in the Data-layer map below, anchored `\bFlooringX\b` / `FlooringX(?![A-Za-z])`.
- ⚑ **Commit granularity — atomic.** schema + migration + code + regen in ONE commit; typecheck is green only when ALL of it is done, so a split leaves a red intermediate. Atomic is safer.
- ⚑ **Shared-dev-DB migration timing.** The dev-N worktrees share ONE database. Renaming 14 tables + 5 enum types is a HARD cutover — sibling branches still referencing `flooring_*` break the instant the migration runs, until they merge this change. Coordinate the deploy window with the user; the user runs the migration.
- ⚑ **GAP — migration object names are DERIVED, not observed.** The index/constraint/sequence names below are derived from Prisma's naming rules (`{table}_pkey`, `{table}_{field}_key`, `{table}_{col}_idx`, `{table}_{col}_fkey`, named indexes = the `map:` label). The canonical file was hand-verified against the live DB. VERIFY every name against the live DB (`\d "flooring_product"` via psql, or `prisma migrate diff`) before finalizing — an out-of-band-renamed index or the autoincrement sequence may not match naive derivation.
- ⚑ **GAP — `import_entry.importNumber` uses `@default(autoincrement())` (schema line ~409), not a `dbgenerated` sequence string.** Its backing sequence is Prisma/Postgres-named `flooring_import_entry_importNumber_seq`. Confirm the actual name and whether a rename is needed (cosmetic; OID-tracked so functionally safe either way).
