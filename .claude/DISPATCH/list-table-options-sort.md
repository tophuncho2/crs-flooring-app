## Intent for this session
Add a `TableOptions` control to the shared list-view `DataTable` engine: an icon trigger rendered inside the leading open-gutter column header that opens a **tabbed** AnchoredPanel popover, with an open-ended tab API (array of `{ key, label, render }`). Move the existing toolbar `SortMenu` body into it as the first tab ("Sort"). Rewire the Work Orders list to use it (drop the toolbar `SortMenu`). Then bring the **inventory list** up to the same multi-column sort standard Work Orders already has (opt-in `maxSortLevels: 3`, swap inventory's single-sort `orderBy` for an ordered entries loop) and surface the same table-options/sort menu into the **inventory-options grid** (the picker grid used by the WO-create adjustment modal). Everything is additive/opt-in: any consumer that doesn't pass `tableOptions` / `maxSortLevels` renders byte-identical to today. "Done" = WO and inventory both sort multi-column through the gutter menu, the WO toolbar `SortMenu` is gone, the WO-create adjustment grid has the same sort affordance, untouched lists are unchanged, and `/check` is green.

## ⚑ Flags — decisions to make / potential gaps
- ⚑ **Where TableOptions chrome lives + folder shape.** Engine convention says mirror the canonical engine structure. Sort already lives at `apps/web/engines/list-view/toolbar/sort/`. The new control is anchored in the *table header* (not the toolbar), so it most naturally lives under `table/` (e.g. `table/options/table-options.tsx` + `table/options/contracts/table-options-contract.ts` + `index.ts`). Decide: new `table/options/` folder vs. a `toolbar/options/` folder. Whatever you pick, export the type + component from the relevant barrel and re-export through the engine root `index.ts` (currently `export * from "./table"` + `export * from "./toolbar"`).
- ⚑ **Popover chrome: pick ONE and be consistent.** Two patterns exist. (a) `AnchoredPanel` (`@/engines/common`) — portal-based, controlled `open`/`onClose`, `trigger` + `stickyHeader` + scrolling body, used by the header filter funnel (`data-table-header-cell.tsx:99-126`). (b) the toolbar `SortMenu`'s bespoke `relative`/`absolute` panel + manual outside-click effect (`sort-menu.tsx:63-126`). The handoff leans toward AnchoredPanel (it's already the header-funnel chrome, so the gutter header stays consistent). AnchoredPanel owns open-state via the consumer, portals to `document.body` (escapes the narrow gutter `<th>` overflow), and provides outside-click/Escape for free. Recommend AnchoredPanel; confirm.
- ⚑ **Extract SortMenu body from its trigger.** `SortMenu` (`sort-menu.tsx`) today bundles its own trigger button + open-state + bespoke panel. For the tab, you want only the *body* (the level rows + Add-level + Clear). Decide: extract a `SortMenuBody` (props `options`/`value`/`maxLevels`/`onChange`) that the Sort tab renders, and either (a) keep `SortMenu` as a thin wrapper for any remaining toolbar consumer, or (b) retire `SortMenu`'s trigger entirely. NOTE: after the WO rewire, `SortMenu` has **no other consumers** (`grep` it) — so retiring the standalone trigger is clean, but the engine test `apps/web/tests/engines/list-view/sort-menu.test.tsx` exercises the `SortMenu` trigger ("Sort" button + badge); update or repoint that test to the new body/menu.
- ⚑ **Gutter is narrow + only renders when `hasOpenColumn`.** The open-gutter `<th>` only renders when `hasOpenColumn` = `Boolean(onOpenRow) || hasRowActions` (`data-table.tsx:194,236-240`); width is 44px (`DEFAULT_OPEN_WIDTH`) or 88px with row actions (`OPEN_WIDTH_WITH_ACTIONS`). Both WO and inventory pass `onOpenRow` + `rowActions`, so they have the gutter. Decide TableOptions degradation when a consumer passes `tableOptions` but there is NO gutter (no `onOpenRow`/`rowActions`): either (a) force-render the gutter `<th>` when `tableOptions` is set (so the trigger has a home — likely the right call, low risk), or (b) no-op. The WO-create grid (`inventory-options-grid.tsx`) uses `onRowClick`, NOT `onOpenRow`, and passes no `rowActions` — so it currently has **no gutter at all**. To surface table-options there you must make `tableOptions` independently force the gutter header (option a). Settle this; it shapes the `hasOpenColumn` / header-render condition.
- ⚑ **Icon + a11y for the trigger.** Icon-only (gutter is 44–88px wide). Pick an icon (e.g. `SlidersHorizontal` / `Settings2` from lucide — `ArrowDownUp` is the current Sort glyph). Give it an `aria-label` ("Table options"). Confirm it visually fits beside the per-row open/options buttons WITHOUT widening the gutter for every row (the trigger sits in the header `<th>` only, not per-row, so it shouldn't affect row gutter width — verify).
- ⚑ **Active-state badge on the trigger.** SortMenu shows a count badge when sorts are active (`sort-menu.tsx:118-122`). Decide whether the TableOptions trigger surfaces an aggregate "active" dot/badge (e.g. when any tab has active state). For now the only tab is Sort; a simple "sorts.length" badge mirrors today. Keep it tab-agnostic if cheap (each tab can report `active?: boolean`), but don't over-build for one tab — note the extraction trigger and move on.
- ⚑ **Inventory backend: nullable/non-default sort fields.** Inventory's sortable fields are `createdAt`, `location` (nullable → `nulls: "last"`), `stockBalance` (sorts on generated `stockQuantity` column) — see `buildInventoryListViewOrderBy` (`read-repository.ts:518-535`). When you convert to a multi-entry loop you must preserve these per-field mappings (mirror WO's `workOrderFieldOrderBy` switch + `appendUniqueOrderBy` dedupe + `createdAt`/`id` tiebreak in the highest-priority direction). Decide the field→orderBy map for inventory and the tiebreak direction rule (WO uses `entries[0]?.direction ?? "asc"`).
- ⚑ **Inventory API validator currently single-sort only.** `apps/web/app/api/inventory/_validators.ts:249-250` parses only `sort`/`sortField` (zod enum). The inventory list request (`data/list-inventory-request.ts:80-94`) likewise reads only `sort`/`sortField`. For multi-sort you must add a `sorts=field:dir,...` parse path on BOTH the request builder/parser AND the API validator — MIRROR the WO validator's `parseSortsParam` + `WORK_ORDERS_UI_SORT_FIELDS` allow-list (`work-orders/_validators.ts:319-338,402-405`). The inventory allow-list is `["createdAt","location","stockBalance"]`. The API route is technically OUTSIDE your owned envelope (Out lists `apps/web/app/api/...` only implicitly) — **FLAG: confirm with the user whether the inventory `_validators.ts` + the GET route belong in your envelope**, because without them the server ignores `?sorts=` and multi-sort silently degrades to single. (The handoff scoped you to inventory's "list backend"; the validator is the API edge of that backend. Treat it as in-scope unless told otherwise, but call it out.)
- ⚑ **Inventory list request `buildInventoryListSearchString` round-trip.** It currently emits only `sort`/`sortField` (`list-inventory-request.ts:96-119`). The engine controller, when `maxSortLevels > 1`, drives a `?sorts=` URL param and sends `input.sorts`. The inventory `listInventoryRequest` builds the query string from `input.sort`/`input.sortField` — so you must teach `buildInventoryListSearchString` to emit `sorts=` from `input.sorts` (mirror nothing — WO's request file is the analog). Decide: emit `sorts` when present, else fall back to the legacy single pair (keeps old bookmarks working, matches WO).
- ⚑ **`InventoryListViewSort` shape change is multi-entry.** The application use case (`list-inventory.ts:82-87`) passes `input.sort` straight through as `{ field, direction }`. The repo's `InventoryListViewSort` is single (`read-repository.ts:389-399`). Decide: introduce `InventoryListViewSortEntry[] { entries }` (mirror `WorkOrdersListSort`) OR keep `sort` single and add `sorts`. WO chose `{ entries: [...] }` end-to-end. Mirror it: use case takes `input.sorts ?? (input.sort ? [input.sort] : [])`, slices to MAX, passes `{ entries }` to the repo. Cap = 3.
- ⚑ **Inventory-options grid is the ONLY consumer of the grid sort.** The handoff says "surface table-options/sort into the inventory-adjustment CREATE modal grid." Validated: `inventory-adjustment-create-modal.tsx` (the record-view shell) is **locked** (no grid — `AdjustmentInventoryIdentityRow` only). Only `work-order-adjustment-create-modal.tsx` renders `InventoryOptionsGrid`. The grid's sort is LOCAL state in `use-inventory-options-grid.ts:76-79,114-121` (single `{field,direction}` + header-click `setSort`). To give it the multi-sort menu you'd add a `sorts`/`setSorts` (or extend `sort`→`sorts`) in that controller and feed the TableOptions config to its `<DataTable>` (`inventory-options-grid.tsx:113-123`). Decide: does the grid get the SAME multi-col standard, or just gain the gutter TableOptions trigger hosting a single-level sort tab? The grid fetches through `listInventoryRequest`, so multi-sort there needs the same request/validator `sorts` path as the list — it rides for free once the list backend is multi-sort. Settle scope: full multi-col in the grid vs. trigger-only.
- ⚑ **Adjustment-create-modal test.** `apps/web/tests/modules/inventory/inventory-adjustment-create-modal.test.tsx` exists and exercises the modal/grid. Re-run + adjust if the grid's `DataTable` props change (e.g. new `tableOptions`).
