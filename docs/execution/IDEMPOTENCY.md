# Idempotency

> **Scope:** Four layers of deduplication that prevent duplicate mutations across the system.
> **Status:** Active

## Rules

1. Every mutation route must participate in at least one idempotency layer.
2. HTTP mutation receipts are the primary layer for API routes.
3. Receipts expire after 24 hours — clients must not replay mutations older than that.
4. Idempotency is enforced before the transaction boundary, not inside it.

## Four Layers

### Layer 1: HTTP Mutation Receipts

The primary idempotency mechanism for API routes.

| Aspect | Detail |
|--------|--------|
| **Scope** | Per user, per scope (route identifier), per idempotency key |
| **Window** | 24 hours from first receipt |
| **Storage** | `AppMutationReceipt` table in PostgreSQL |
| **Enforcement** | `enforceMutationReceipt()` in `route-policy.ts` |
| **Client contract** | Every mutation request includes `mutation.idempotencyKey` in the body |

Flow:
1. Client generates a UUID idempotency key per mutation attempt.
2. `parseMutationEnvelope()` extracts the key from the request body.
3. `enforceMutationReceipt()` checks for an existing receipt with the same key.
4. If found: return the cached response (short-circuit).
5. If not found: reserve a receipt, execute the mutation, finalize with the response.

### Layer 2: Domain-Level Deduplication (sourceVersion)

Work order allocation use cases check `sourceVersion` to detect stale requests:

- Client sends `expectedUpdatedAt` timestamp.
- Server compares with actual `updatedAt` via `assertExpectedUpdatedAt()`.
- Mismatch returns 409 Conflict — client must refetch and retry.

This is optimistic concurrency control, not strict idempotency, but it prevents duplicate effects from stale state.

### Layer 3: BullMQ Job ID Deduplication

Jobs dispatched to BullMQ use deterministic job IDs generated by `toBullMqJobId()`:

- Same outbox event always produces the same job ID.
- BullMQ rejects duplicate job IDs automatically.
- Prevents the relay from double-dispatching the same event.

### Layer 4: Database Unique Constraints

The final safety net. Prisma unique constraints prevent duplicate records even if all other layers fail:

- Email uniqueness on User model.
- Composite uniqueness on domain records where applicable.
- Caught by `normalizePrismaError()` and returned as 409 responses.

## Patterns

Client-side mutation assembly (shared engine):
```
modules/shared/engines/common/transport/mutations.ts
  └── withMutationMeta()  ← Attaches idempotencyKey + scope to request body
```

Server-side receipt lifecycle:
```
route-policy.ts
  ├── parseMutationEnvelope()      ← Extract payload + meta
  ├── enforceMutationReceipt()     ← Check/reserve receipt
  └── finalizeMutationReceipt()    ← Cache response for replay
```

Database:
```
packages/db/
  └── mutation-receipts.ts
      ├── reserveAppMutationReceipt()
      ├── getAppMutationReceipt()
      └── finalizeAppMutationReceipt()
```

## Anti-Patterns

1. **Do not** skip mutation receipts for POST/PATCH/DELETE routes.
2. **Do not** generate idempotency keys server-side — the client owns the key.
3. **Do not** rely solely on database constraints for idempotency — they are the last resort.
4. **Do not** cache mutation receipts beyond 24 hours — stale receipts must expire.
